Task 1 - Prepare logging environment

  1. Choose and deploy your monitoring stack. The default choice is:
  • logstash is a server-side data processing engine that receives data from multiple sources, parses the log, and then sends it to the Elasticsearch database
  • elasticsearch is the core of the entire system, which combines the functions of a database, search engine, and analytical system
  • kibana allows users to visualize data using charts and graphs in Elasticsearch. You can also administer the database through Kibana
  • filebeat agents are the logging agent installed on the machine generating the log files
  1. Try to deploy different roles within your cluster: coordinator, master node, data node…
  2. Configure basic indices politics: shards allocation politics, numbers of primary & replica shards, JVM heap usage configuration…

Keep in mind that usually, the ELK cluster requires quite high system resources.

Then I set up initial master node option. During the initial loading of the cluster, the nodes participating in the voting on the choice of the master are determined, so in the new cluster we must specify a list of these nodes.

There is a jvm.options file in the Elasticsearch installation directory, which already specifies the heap sizes, by default it is 1 GB. I give 2 GB to each machine so 1 GB is enough, because according to Elastic documentation, it should not exceed 50% of the node’s memory.

Then I turn off paging because it negatively affects the performance and stability of Elasticsearch

Then I configured file descriptors. Elasticsearch works with a large number of file descriptors, and their shortage can have a disastrous effect on its work. According to the official documentation, it is necessary to specify the value of file descriptors at least 65,536.

Bonus 2: configure and test extended index management politics: clean old logs, backup logs, time period to create & update indexes…

We started by changing the elasticsearch configuration by adding a path - path.repo: /etc/backup. Then we used it to create a repository (Figure 24).


Figure 24 - Repository

And then we created policies as shown below:

PUT _slm/policy/snaps
{
  "schedule": "0 10 17 * * ?",       
  "name": "<snaps-{now/d}>", 
  "repository": "backup",    
  "config": {
    "indices": "*",                 
    "include_global_state": true    
  },
  "retention": {                    
    "expire_after": "30d",
    "min_count": 5,
    "max_count": 50
  }
}

It will create a backup of the cluster state, all data streams, and all indices daily at 20:10 UTC+3. This configuration keeps snapshots for 30 days, retaining at least 5 and no more than 50 snapshots regardless of age.
We can see the new policy at Stack Management>Snapshot and restore>Policies (Figure 25).


Figure 25 - New policy
